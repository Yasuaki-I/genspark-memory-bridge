## 作業ログ

### 1. 通信エラーの原因切り分け

PowerShell の Select-String で background.js の通信仕様を調査。以下が判明:

- background.js は `chrome.tabs.sendMessage(tabId, { type: "GMB_BUILD_MARKDOWN" })` で content_script に問い合わせる
- レスポンスは `{ ok: true, data: { markdown, meta } }` を期待
- 一方 content_script.js は `action` キーで `extractChat` / `previewFilename` / `buildMarkdown` の3つに応答する設計
- → キー名（type vs action）と値（GMB_BUILD_MARKDOWN vs buildMarkdown）が両方ミスマッチ

### 2. 修正方針決定

background.js は T1-5 で動作実証済の安定版なので触らず、content_script.js 側にエイリアスリスナー（`type: "GMB_BUILD_MARKDOWN"` を受けて既存 `buildMarkdown()` を呼ぶ）を追加する方針に決定。

### 3. エイリアスリスナー追加 → 構文エラー発生

content_script.js 末尾のリスナー内に14行追加（363行に）。しかし拡張機能リロード後にカードに「エラー」バッジが点灯。

DevTools で Console を確認すると:
```
Uncaught SyntaxError: Invalid or unexpected token (at content_script.js:354:9)
```

### 4. 真の原因判明

354行目を確認したところ、AI が直前のターンで「説明用の図解」として書いた装飾注釈文字列が、コードブロックの一部としてユーザー側にコピペで取り込まれていた:

```
});   ← ★この閉じ括弧はそのまま残す（移動しない）
```

`← ★…` の部分が JavaScript として解釈不能 → ファイル全体の実行中断 → `__GMB_*` 関数が未定義 → 通信失敗、という連鎖だった。

### 5. PowerShell 正規表現置換で安全に修正

```powershell
$content = Get-Content content_script.js -Raw
$fixed = $content -replace '\}\);\s+← ★この閉じ括弧はそのまま残す（移動しない）', '});'
Set-Content -Path content_script.js -Value $fixed -NoNewline
```

修正後、`Select-String -Pattern "←|★"` で残存ゼロを確認。

## 発見・学び

- AI が説明用に書いた装飾注釈がコードブロックに含まれていると、ユーザーがコピペしたときに構文エラーの原因になる。コードブロック内には実コードのみを置くこと。引き継ぎドキュメントに「装飾注釈の禁止」ルールを新設。
- chrome://extensions の「エラー」バッジは `console.warn` も拾う仕様で、必ずしも致命的とは限らない。Service Worker の DevTools と Console コンテキスト切替後の `__GMB_*` 関数チェックで実際の状態を確認するのが確実。
- 通信エラーの原因切り分けは「background.js の sendMessage 仕様 → content_script.js のリスナー仕様」の順で PowerShell の Select-String を走らせると確実に特定できる。Phase 2 以降の通信系トラブル時にも有効なパターン。
