## 結論

Genspark チャットの AI 応答 HTML → Markdown 逆変換に turndown ライブラリ（v7.2.0）を採用する。自前軽量変換器へのフォールバックは行わない。

## 検証内容

Genspark の対象チャット（GensparkMemoryBridge_002）上で以下を実施。

1. CDN（unpkg）経由で turndown@7.2.0 を動的ロード → 成功（typeof TurndownService === 'function'）
2. 最小サンプル `<h2>テスト</h2><p>本文 <strong>太字</strong></p>` を変換 → 期待通りの出力
3. 実チャットの AI 応答（IDX=1、h2 / ol / ul / strong / pre>code / hr を含む網羅性の高いメッセージ）を変換 → 4要素すべて正しく保持

### 4要素保持の確認結果

| 要素 | 結果 |
|---|---|
| 見出し（h2） | `## ...` で正しく変換 |
| 太字（strong） | `**...**` で正しく変換 |
| 順序付きリスト（ol） | `1. ...` で正しく変換 |
| 順序なしリスト（ul） | `- ...` で正しく変換 |
| コードブロック（pre>code） | フェンス形式で言語指定（javascript）も保持 |
| 区切り線（hr） | `* * *` で正しく変換（Markdown 仕様準拠） |
| インラインコード | バッククォート形式で正しく変換 |

## 採用理由

1. 正確性: 見出し・コードブロック・リスト・太字すべて期待通りに変換され、追加実装不要
2. 言語指定の保持: `<pre><code class="language-javascript">` の言語情報がフェンス付き Markdown コードブロックの言語指定として保持される（コードブロックの可読性に直結）
3. Manifest V3 互換: ピュア JavaScript・eval 不要で Chrome 拡張に同梱可能
4. 設定の柔軟性: headingStyle / codeBlockStyle / bulletListMarker / emDelimiter で出力スタイルを制御可能
5. CSP 制約なし: Genspark のページ上で外部スクリプトの動的注入が許可されている（検証中のみ。本実装ではバンドル）

## 採用に伴う制約と対応方針

### 制約1: コードブロックに UI の「Copy」テキストが混入する

Genspark の UI は `<pre>` 内にコピーボタン相当のテキスト「Copy」を埋め込んでいる。turndown はそれをコードブロックの本文として取り込んでしまう。

対応方針: turndown 変換前に DOM の前処理を行う。具体的には、`<pre>` 配下のコピーボタン要素（および本文以外のノード）を除去してから turndown に渡す。実装は T1-3（content_script.js 抽出ロジック）で組み込む。

実装スケッチ:

~~~javascript
// 変換前の前処理
function sanitizeViewer(viewer) {
  const clone = viewer.cloneNode(true);
  // pre 内の本文以外（コピーボタン等）を除去
  clone.querySelectorAll('pre').forEach(pre => {
    pre.querySelectorAll('button, .copy-btn, [class*="copy"]').forEach(el => el.remove());
  });
  return clone.innerHTML;
}
~~~

実際の Genspark の DOM 構造に合わせて T1-3 で詳細を詰める。

### 制約2: data-source-line 属性は出力に残らない

これは Markdown 仕様上の制約であり、turndown の挙動として期待通り。問題なし。

### 制約3: 検証中のスコープ管理

DevTools コンソールで `const td = ...` と宣言したインスタンスがコンテキストリフレッシュで失われる事象を確認。検証中は `window._td = new TurndownService(...)` 形式で永続化推奨。本実装（content_script.js）ではモジュールスコープで管理するためこの問題は発生しない。

## 代替案と却下理由

### 代替案A: 自前軽量変換器（HTML タグを正規表現で Markdown 記法に置換）

却下理由: turndown が問題なく動作したため、車輪の再発明をする必要がない。ネストしたリスト・コードブロック内の特殊文字エスケープ等、自前実装ではエッジケース対応のコストが高い。

### 代替案B: marked などの双方向ライブラリ

却下理由: marked は Markdown → HTML の変換ライブラリで方向が逆。本用途には turndown が直接的に適合する。
