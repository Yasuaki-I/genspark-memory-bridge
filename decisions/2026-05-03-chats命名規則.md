# chats/ 命名規則決定（T1-2 結果）

- 判断日: 2026-05-03
- 判断者: チャット GensparkMemoryBridge_002
- 関連タスク: tasks/phase1-tasks.md T1-2
- ステータス: 確定

## Genspark側のチャット命名前提（本決定の基礎）

本決定は Genspark のチャット命名が以下の規則で運用されることを前提とする。命名規則を変更する場合は本決定も見直す必要がある。

- 形式: {プロジェクト名}_{連番3桁}
- プロジェクト名: アンダースコアを含まない（キャメルケース）
- 連番: 3桁ゼロパディング固定（_001, _002, ..., _999）
- 例: GensparkMemoryBridge_002, OpenClaw_001, KADOKAWA_005

判定用正規表現: `^([A-Za-z0-9]+)_(\d{3})$`

正規表現にマッチしない場合の扱い: フォールバックとしてチャット名そのままをプレフィックスとして扱い、連番なしファイルとして保存する（後述の論点1代替仕様）。

## 決定事項

### 論点1: ファイル名規則

採用: chats/{プロジェクト名}/{チャット名}_{最終保存日}.md

例:
- chats/GensparkMemoryBridge/GensparkMemoryBridge_002_2026-05-04.md
- chats/OpenClaw/OpenClaw_001_2026-05-10.md

理由:
- プロジェクト単位でフォルダ分けすることで、ファイラーでの一覧性が高い
- Phase 2 の復元機能で「同プロジェクトの過去ログを最新順に取得」する際、対応するフォルダのみを見れば済む
- OpenClaw 側の Obsidian 運用（プロジェクトごとに ★OpenClaw\、★GensparkMemoryBridge\ で分ける発想）と一貫している

代替仕様（命名規則にマッチしない場合）:
- 形式: chats/_unsorted/{チャット名}_{最終保存日}.md
- 例: chats/_unsorted/2026-05-03_自由形式チャット_2026-05-03.md
- _unsorted フォルダはチャット名規則を後付けで決めかねるケース用の退避先

### 論点2: 同名チャットが複数日にまたがる場合

採用: 最新保存日のファイルのみ保持（古いファイルは削除して新規作成）

例:
- 5/3 に GensparkMemoryBridge_002 を保存 → chats/GensparkMemoryBridge/GensparkMemoryBridge_002_2026-05-03.md 作成
- 5/4 に同チャットを再保存 → 旧ファイル削除 → chats/GensparkMemoryBridge/GensparkMemoryBridge_002_2026-05-04.md 新規作成

理由:
- Genspark のチャットは「途中保存」しても DOM に全履歴が残っている。すなわち保存ボタン押下時点で常に「そのチャットの全履歴のスナップショット」が取得できる
- したがって最後に保存したファイルが最も完全な状態。古い日付のスナップショットは情報量が少ないだけで価値がない
- 過去版が必要になった場合は GitHub の commit 履歴から取得可能（rollback 可能）

実装上の要件:
- 保存処理の前に、対象チャット名（連番含む）と一致する既存ファイルを chats/{プロジェクト名}/ 配下から検索し、見つかれば削除する
- 検索条件: ファイル名のプレフィックス `{チャット名}_` で始まるもの全件

### 論点3: 既存ファイルがある場合の挙動

採用: 常に上書き（確認ポップアップなし）

例:
- 同日に同チャットを2回保存 → ファイル名が同一になる（YYYY-MM-DDが同じ）→ 後勝ちで上書き

理由:
- Genspark のスナップショット保存は冪等操作。同じチャットを何度押しても同じ最終状態が取れる
- 確認ポップアップは MVP では不要（ユーザー本人のみが使う、誤操作リスク低）
- GitHub 側の commit 履歴で過去版は完全に追跡可能（GitHub API で PUT する際に sha を渡せば既存ファイル上書きとして commit が残る）

実装上の要件:
- GitHub API: GET /repos/{owner}/{repo}/contents/{path} で既存ファイルの sha を取得
- GitHub API: PUT /repos/{owner}/{repo}/contents/{path} に sha を含めて送信（新規作成時は sha 不要）

## 後続タスクへの影響

### T1-3（content_script.js 抽出ロジック実装）

- document.title から取得したチャット名を正規表現 `^([A-Za-z0-9]+)_(\d{3})$` で分解する処理を追加
- マッチした場合: プレフィックス（プロジェクト名）と連番を分離 → ファイルパス生成に使用
- マッチしない場合: _unsorted フォルダにフォールバック保存

### T1-5（GitHub API 呼び出し実装）

- 保存前に同チャット名の既存ファイル検索 → 削除 → 新規作成、の3ステップを実装
- 既存ファイル検索は GET /repos/{owner}/{repo}/contents/chats/{プロジェクト名}/ でディレクトリ一覧を取得し、`{チャット名}_` で始まるファイルを抽出する方式が妥当
- ディレクトリが存在しない場合（初回保存時）は404になるため、エラーハンドリングを実装

### Phase 2（復元機能）

- 同プロジェクトの過去ログを最新順に取得する際、chats/{プロジェクト名}/ 配下のファイル一覧をファイル名末尾の日付（YYYY-MM-DD）でソートする
- 「直前のチャット」の特定は、現在のチャット連番から1を引いた連番のファイルを検索する方式が候補（例: _002 を開いている場合、_001 を探す）

## 代替案と却下理由

### 代替案: ファイル名を {チャット名}.md（日付なし）にする

却下理由: 同名チャットの上書き判定はシンプルになるが、ファイル一覧で「いつ保存されたか」が見えなくなる。GitHub の commit 日時で確認することは可能だが、ローカルクローンや Obsidian 上での確認が手間になる。

### 代替案: フォルダ分けせず chats/ 直下にすべて配置

却下理由: プロジェクト数が増えた際の一覧性が悪化する。Phase 2 で「同プロジェクトの過去ログ」を絞り込む際にもフォルダ分けされている方が処理が単純。

### 代替案: 保存時に古いファイルを削除せず履歴として残す

却下理由: GitHub の commit 履歴で過去版は追跡可能なため、ファイル単位での履歴保持は冗長。複数ファイルが残ると「どれが最新か」の判定が必要になり、Phase 2 の処理が複雑化する。

## 確認事項（将来見直し）

- プロジェクト数が10を超えた場合、_unsorted フォルダの運用見直しが必要になる可能性
- 連番が3桁を超える場合（_1000以上）の扱い → 現実的にはほぼ発生しないが、Phase 4 以降で必要に応じて4桁対応を検討
