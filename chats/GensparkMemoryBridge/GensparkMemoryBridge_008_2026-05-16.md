## プロジェクト概要

GensparkのAIチャットにメモリ永続化の仕組みを構築するプロジェクト。Chrome拡張機能でチャット内容のGitHub保存・新チャットへの復元を実現する。

### ゴール（完成状態の定義）

1. 保存フロー: Gensparkチャット終了 → Chrome拡張のボタン押下 → 会話全文がMarkdown化されGitHub Repoにcommit
2. 復元フロー: 新チャット開始 → Chrome拡張がチャット名パターン（OpenClaw_00x等）を検知 → GitHubから前回ログ＋引き継ぎドキュメントを取得 → クリップボードにコピー → 手動ペースト（MVP）
3. ドキュメント統合: ObsidianドキュメントがGitHub経由で同期されている

## 技術スタック

- Chrome拡張機能（Manifest V3）
- GitHub API
- Obsidian + Obsidian Git

## 通信仕様メモ（5/10時点）

content_script.js が応答するアクション:

- `extractChat`（action キー）→ {success, markdown, messageCount, meta}
- `previewFilename`（action キー）→ {ok, filename, filePath, chatTitle, messageCount}
- `buildMarkdown`（action キー）→ {ok, data}
- `GMB_BUILD_MARKDOWN`（type キー・background互換エイリアス）→ {ok, data}

background.js は `chrome.tabs.sendMessage(tabId, {type:"GMB_BUILD_MARKDOWN"})` で問い合わせ、`{ok:true, data:{markdown, meta}}` を期待する。

background.js のエラーマッピング:

- `mapGitHubError(response, errorBody)`: HTTPステータスを日本語メッセージに変換
- `mapNetworkError(error)`: TypeError（fetch失敗）を「ネットワーク接続を確認してください」に変換、それ以外は元メッセージを返す
- `safeJson(response)`: response.json() を安全に取得、失敗時は空オブジェクト
- `checkSettings(settings)`: PAT/repo/user の欠落を識別し具体的な未設定項目を含むメッセージを返す

### バックアップ運用ルール【5/10改訂・選択肢C採用】

手元バックアップの保持方針:

直近タスクの完了時点バックアップを2世代まで保持する。具体的には「現在着手中タスクの直前タスク完了版」+「1つ前のタスク完了版」の合計2世代を手元に保持する。これより古いバックアップは削除し、必要時は GitHub のコミット履歴から取得する。

理由:

ロールバック先として現実的なのは直近1〜2タスク前まで。手元バックアップは即時復元用、GitHub は長期履歴用、と役割分担する。手元バックアップがあればネット切断時・GitHub障害時・git操作不慣れな状況下でも、ファイルを上書きコピーするだけで即座に復元できる。

命名規則:

`{ファイル名}.bak_{YYYYMMDD}_{次タスクID}_pre`

例: `background.js.bak_20260510_t18_pre` は T1-7 完成時点（T1-8 着手直前）のバックアップ。日付は次タスク着手日、タスクIDは次タスクのID。

整理タイミング:

各タスク完了時に、新しいバックアップを取得した直後に「2世代前」を削除する。整理コマンドは引き継ぎドキュメントに記載せず、毎回チャットで案内する（非エンジニア配慮）。

Phase 完了時:

Phase 1 完了判定（T1-10）後に、Phase 1 全体の最終版バックアップ1セットを `bak_phase1_final` として保持し、それ以外を削除。Phase 2 開始時のロールバック先を確保する。

## 1\. ご理解の正しさについて

**結論: 半分正しく、半分は注意が必要**です。

### 正しい部分

「Claude で memory.md をクラウド環境に置いて読みに行く設定をしていれば、記憶の永続化が理論上可能」 — これは**Claude Desktop + MCP (Model Context Protocol)** または **Claude Projects のシステムプロンプト/ナレッジ機能**を使えば、概ね実現可能です。

具体的には:

-   **Claude Projects 方式**: プロジェクトのカスタムインストラクション（システムプロンプト）に「あなたの記憶は memory.md に格納されています」と書き、ナレッジに memory.md をアップロードしておけば、新規チャット開始時に Claude は自動的にその内容を参照します。ペルソナ設定も同じ仕組みで永続化できます。
-   **MCP 方式**: Claude Desktop に GitHub MCP サーバーや Filesystem MCP サーバーを接続すれば、Claude がチャット中に memory.md をリアルタイムで読み書きできます。これは「読み込み」だけでなく「書き込み（記憶の更新）」も自動化できる点で、Projects 方式より強力です。

### 注意が必要な部分

「永続化」と一口に言っても、実は3つのレベルがあります。

レベル

内容

実現難易度

L1

新チャット開始時に過去の記憶を**読み込む**

容易（Projects で十分）

L2

チャット中に発生した新しい記憶を**自動的に書き込む**

中（MCP 必要）

L3

チャット終了時に**自動的にサマリ化して保存**

難（自動トリガーが課題）

L1 は Projects のナレッジ機能だけで実現できます。L2 は MCP + Claude Desktop が必要です。L3 は完全自動化が難しく、現実的には「セッション終了時にユーザーが明示的に依頼する」運用になります。

GensparkMemoryBridge が解決しようとしているのは、まさにこの L3 を**Genspark 上で**実現する仕組みです。

## 2\. GensparkMemoryBridge への適用可能性

「GensparkMemoryBridge でも記憶とペルソナの永続化ができるのでは」というご質問への答えは **「保存側は YES、復元側（自動読み込み）は限定的に YES」** です。

### 保存側（YES）

すでに Phase 1 で実装中の機能（チャット内容の Markdown 化 → GitHub コミット）は、そのまま「記憶の蓄積」として機能します。memory.md のような単一ファイルではなく、`chats/{プロジェクト名}/{連番}_{日付}.md` という時系列ログとして蓄積される形式ですが、これは**むしろ memory.md より情報量が豊富**です。

ペルソナ設定についても、専用ファイル（例: `personas/{プロジェクト名}.md`）を GitHub 上に置いて、拡張機能から読み書きできるようにすれば実現可能です。Phase 1 の API 呼び出しコード（saveToGitHub / readFromGitHub）がほぼそのまま流用できます。

### 復元側（限定的に YES）

ここが Genspark 固有の制約と向き合う必要がある部分です。

Claude Projects は「システムプロンプト」「ナレッジ」「カスタムインストラクション」という形で、**チャット開始前に AI へ情報を渡す公式の仕組み**を提供しています。一方、Genspark にはこれに相当する公式機能が**ない**（少なくとも私の知る限りでは）。

そのため GensparkMemoryBridge の Phase 2 復元方式は「クリップボードにコピー → 手動 Ctrl+V ペースト」という MVP 設計になっています。これは技術的制約からくる妥協です。

つまり Genspark で実現できる復元は:

-   ✅ チャット冒頭にユーザーが手動で過去ログ + ペルソナをペースト（MVP）
-   ✅ 拡張機能がチャット名パターン検知 → 自動でクリップボードに準備（Phase 2）
-   ❌ 完全自動注入（Genspark の DOM 操作で input 欄に挿入することは技術的には可能だが、Genspark 側の仕様変更で壊れやすく、Phase 4 後回し）
-   ❌ チャット開始前のシステムプロンプト相当（Genspark に該当機能なし）
