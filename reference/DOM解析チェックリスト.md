# Genspark DOM 解析チェックリスト

5/2 の T0-3 で確認する観点。Go/No-Go 判断はこのチェック結果に基づく。

## A. セレクタ安定性（最重要・Go判断の主基準）
- [ ] チャット全体を包むルートコンテナに data-* 属性があるか
- [ ] 個々のメッセージブロックに data-message-id 等の一意識別子があるか
- [ ] role="log" / role="article" / role="listitem" 等の ARIA role が付与されているか
- [ ] aria-label / aria-live でメッセージ種別が判別できるか
- [ ] クラス名がハッシュ化（例: css-1a2b3c4）されているか、意味のある名前か

## B. メッセージ構造
- [ ] ユーザー発話と AI 応答を区別する属性・クラスは何か
- [ ] メッセージ本文（テキスト）はどの要素に格納されるか
- [ ] コードブロック・表・画像はどの要素・どの形式で保持されるか
- [ ] Markdown 由来の構造（見出し・リスト等）は HTML として残っているか

## C. 動的挙動
- [ ] ストリーミング中（AI回答生成中）の DOM 変化パターン
  - テキストノードへの追記か、innerHTML 全置換か
- [ ] 完了判定に使えるシグナル（属性変化・ボタン状態・aria-busy 等）
- [ ] 仮想スクロール / 遅延レンダリングの有無
  - 過去メッセージが DOM から消える挙動があるか
- [ ] スクロールせずに全文取得できるか、強制ロードが必要か

## D. ネットワーク層（No-Go時の代替案A準備）
- [ ] チャット内容を返す API エンドポイントの有無
- [ ] レスポンス形式（JSON / SSE / WebSocket）
- [ ] 認証ヘッダの取得可否

## E. その他
- [ ] チャットタイトル・URL からのチャット識別子取得方法
- [ ] OpenClaw_00x 等のパターン検知に使える DOM 位置

## Go/No-Go 判断基準
- **Go**: A で2項目以上、または B で明確な区別属性が見つかる
- **No-Go**: A・B ともに不安定 → 代替戦略 A/B/C を検討
  - A. ネットワークリクエストインターセプト
  - B. テキストベース抽出（構造を諦める）
  - C. XPath 深度ベース（脆いが最終手段）
